= STARTS Maven Plugin - Fork efluid
:toc: left
:toclevels: 3
:icons: font
:source-highlighter: highlight.js
:sectnums:

Fork du plugin Maven https://github.com/TestingResearchIllinois/starts[STARTS] (Universite d'Illinois) adapte aux besoins efluid : selection statique des tests, support multi-modules, parallelisme, suivi des echecs entre runs (mode RETRY), et outils de debug du graphe de dependances.

== Vue d'ensemble

STARTS analyse statiquement les dependances entre classes via `jdeps`, calcule les checksums des classes compilees, et entre deux runs n'execute que les tests affectes par les classes modifiees.

Principales adaptations efluid :

* Workflow `run-selected` qui enchaine Surefire (TU) + Failsafe (TI), avec patch automatique de `framework2.properties` et initialisation BDD locale
* Support multi-modules : STARTS s'execute sur chaque module du reactor, chacun selectionnant ses tests affectes
* Mode RETRY : tant qu'un module a des tests en echec, seuls ces tests sont rejoues
* Goal `debug-paths` : plus court chemin entre une classe modifiee et chaque test, pour identifier le code glue
* Optimisations de build : exclusion des modules npm, desactivation de la recompilation incrementale agressive

== Installation

=== Build du plugin

[source,bash]
----
cd starts/
mvn clean install -DskipTests
----

IMPORTANT: A relancer apres chaque modification du code du plugin.

=== Verification

[source,bash]
----
mvn help:describe -Dplugin=edu.illinois:starts-maven-plugin:1.5-SNAPSHOT -Dgoal=run-selected
----

NOTE: Les invocations directes en ligne de commande (`mvn groupId:artifactId:goal`) ne lisent PAS le `pluginManagement`. La version `1.5-SNAPSHOT` est donc indiquee explicitement dans tous les scripts. C'est une limitation Maven connue (MNG-4965).

== Scripts

Trois scripts dans `archi/scripts/starts/`. ecore etant un sous-module du reactor archi, il est couvert automatiquement par le mode reactor complet.

=== run-tests.sh

Le workflow principal.

[source,bash]
----
./run-tests.sh                  # tout le reactor (incl. ecore)
./run-tests.sh app              # uniquement le module app
./run-tests.sh ws/archi-rest    # uniquement ce module
MODULE=app ./run-tests.sh       # equivalent via variable
./run-tests.sh app -DsurefireForkCount=8   # options Maven en plus
----

Etapes :

[arabic]
. Compilation des modules + dependances (multi-thread, sans recompilation totale)
. `starts:prepare` (1 fois) : patch `framework2.properties` + init BDD
. `starts:run-selected` : selection + execution des tests, par module
. Agregation des echecs + restauration de `framework2.properties`
. Bilan des temps (compilation / BDD / tests / total)

=== init-graph.sh

Repart sur une base propre : supprime les `.starts/` et recalcule graphe + checksums, sans executer de tests.

[source,bash]
----
./init-graph.sh                 # tout le reactor
./init-graph.sh app             # uniquement app
./init-graph.sh echange         # uniquement echange (utile pour purger un module)
----

A lancer apres un `git pull` important, un changement de branche, ou si le graphe semble incoherent.

=== debug-paths.sh

Analyse le graphe pour identifier pourquoi un test est selectionne. N'execute aucun test.

[source,bash]
----
./debug-paths.sh                              # tous les couples
./debug-paths.sh -DforClass=FiltreRequete     # depuis une classe (substring)
./debug-paths.sh -DforTest=TestFiltreRequete  # vers un test
----

Resultat dans `.starts/debug-paths.txt` :

[source]
----
## com.example.MaClasse -> 3 test(s)
  com.example.MaClasse -> com.example.Service -> com.example.TestService
  com.example.MaClasse -> com.example.TestMaClasse
----

Pour chaque test, le chemin le plus court vers la classe modifiee la plus directement responsable. Un test = une raison principale.

== Mode RETRY (fix & retry)

Les tests en echec sont rejoues automatiquement au run suivant, par module.

[source,bash]
----
./run-tests.sh app
# 50 tests, 3 echecs -> app/.starts/failed-tests.txt cree

# correction du code

./run-tests.sh app
# MODE RETRY : seuls les 3 echecs rejoues (changements courants ignores)
# 1 echoue encore -> failed-tests.txt = 1

./run-tests.sh app
# MODE RETRY : seul le dernier
# vert -> failed-tests.txt supprime, checksums mis a jour

./run-tests.sh app
# retour au mode STARTS normal
----

Comportement : tant que `<module>/.starts/failed-tests.txt` n'est pas vide, on ne rejoue QUE ces echecs. Une fois tout vert, retour a la selection STARTS normale.

NOTE: Le fichier `failed-tests.txt` est LOCAL a chaque module. Le fichier agrege `scripts/starts/work/failed-tests.txt` est en LECTURE SEULE (consultation de tous les echecs d'un coup) ; le modifier n'a aucun effet sur les runs.

Pour forcer un re-run complet : `./init-graph.sh`.

== Parametres importants

Surchargeables via `-D`.

|===
| Parametre | Defaut | Description

| `exclureArchiDep`     | `true`  | Exclut `com.efluid.*`, `com.hermes.arc.*` du graphe jdeps (false pour archi lui-meme)
| `surefireForkCount`   | `4`     | JVM en parallele pour les TU (reuseForks=true)
| `failsafeForkCount`   | `1`     | JVM pour les TI (reuseForks=false, isolation BDD)
| `skipDbInit`          | `false` | Si true, n'initialise pas la BDD (deja fait par prepare en multi-module)
| `skipPropertiesPatch` | `false` | Si true, ne patche pas framework2.properties (deja fait par prepare)
| `workDir`             | -       | Repertoire de travail partage (consultation des echecs agreges)
| `updateChecksums`     | `true`  | Met a jour les checksums apres succes
| `forClass` / `forTest`| -       | (debug-paths) Filtre par substring FQN
| `maxPathsPerClass`    | `1000`  | (debug-paths) Limite de chemins par classe modifiee
|===

== Optimisations de build

Le build complet passait de ~8 min a ~2 min grace a :

`-Dmaven.compiler.useIncrementalCompilation=false`:: Casse le cercle vicieux ou le JAR regenere par `install` etait detecte comme "nouvelle dependance", declenchant une recompilation totale. Avec ce reglage, seuls les fichiers reellement modifies sont recompiles.

Exclusion des modules npm:: Les modules 100% front (`frontend-maven-plugin`) sont exclus du build et de l'analyse via `EXCLUDED_MODULES` en haut du script :
+
[source,bash]
----
EXCLUDED_MODULES='!com.efluid:archi-js-ressources-a-copier,!com.efluid:archi-js-test-serveur-cti,!com.efluid:archi-js-composants-webjar,!com.efluid:archi-frontend'
----

`-Dskip.node=true`:: Pour les modules qui contiennent a la fois du Java testable ET une partie front (ex: `archi-batch-launcher-ui`), on ne peut pas les exclure sans perdre leurs tests. La config des poms efluid branche le `frontend-maven-plugin` sur la property `${skip.node}` : passer `-Dskip.node=true` desactive toutes les etapes npm/node (install, build front) tout en gardant la compilation et les tests Java. C'est preferable a l'exclusion quand le module a du Java utile.
+
En resume : module 100% front -> `EXCLUDED_MODULES` ; module mixte Java+front -> `-Dskip.node=true`.

Auto-skip:: Les modules de type `pom` (agregateurs) et ceux sans `src/test/java` sont ignores immediatement, sans cout.

Compilation deleguee au shell:: Le plugin ne recompile plus (`@Execute(TEST_COMPILE)` retire). La compilation est faite une seule fois par le script, et Surefire est invoque via le goal direct `surefire:test` (pas la phase `test`) pour ne pas relancer le cycle.

== Goals Maven

|===
| Goal | Description

| `starts:clean`        | Supprime `.starts/`
| `starts:diff`         | Classes modifiees + tests affectes. Avec `-DupdateDiffChecksums=true` : reconstruit le graphe
| `starts:prepare`      | Pre-requis globaux (patch properties + init BDD), 1 fois, non-recursif (`-N`)
| `starts:run-selected` | Workflow complet : selection + TU + TI + suivi echecs
| `starts:debug-paths`  | Chemins classe modifiee -> test
|===

== Architecture technique

=== Mojos (starts-plugin)

* `BaseMojo` -> `DiffMojo` -> `RunSelectedMojo` : hierarchie principale
* `PrepareMojo` : pre-requis globaux
* `DebugPathsMojo` : analyse des chemins

=== Runner

* `MavenTestRunner` : execution Surefire/Failsafe (sous-process), decoupage en lots, filtre des tests inexistants
* `TestSelector` : calcul des tests affectes, separation TU/TI
* `PropertiesGuard` : patch + restauration de framework2.properties
* `FailedTestsTracker` : parsing XML, suivi des echecs (failed-tests.txt local)
* `ProgressWatcher` : suivi temps reel des tests
* `PathFinder` : BFS plus court chemin (debug-paths)
* `RunReport` : logs combines console + fichier horodate

=== Fichiers generes

[source]
----
<module>/.starts/
  deps.zlc            # Index inverse classe -> tests dependants
  graph               # Graphe (source destination par ligne), ecrit par init-graph
  selected-tests      # FQN des tests selectionnes
  failed-tests.txt    # Echecs du module (mecanisme RETRY)
  debug-paths.txt     # Sortie debug-paths

scripts/starts/
  work/
    failed-tests.txt  # Agregation LECTURE SEULE (consultation)
  logs/
    <module>-YYYYMMDD_HHMMSS.log
----

== FAQ

=== Un test supprime est encore execute

Un `.class` orphelin peut subsister dans `target/test-classes`, ou un vieux graphe le referencer. Le plugin filtre les tests dont le `.class` n'existe plus, mais pour purger : `./init-graph.sh <module>`.

=== "Surefire plugin not available"

Module de type `pom` (agregateur). Gere par l'auto-skip ; si l'erreur apparait, verifier que le plugin est a jour.

=== La compilation prend toujours 8 min

Verifier que `-Dmaven.compiler.useIncrementalCompilation=false` est bien dans le script et que le plugin est rebuilde. Un second build sans modification doit afficher `Nothing to compile` sur la quasi-totalite des modules.

=== Forcer le relancement de tous les tests

`./init-graph.sh` supprime les `.starts/` : au run suivant, plus rien n'est considere comme modifie. Modifier une classe declenche alors la selection des tests affectes.

== Versionning du debug-paths

`DebugPathsMojo` affiche sa version en console et dans le fichier de sortie (constante `DEBUG_PATHS_VERSION`). Pratique pour confirmer qu'un rebuild a ete pris en compte.
